SET∞

Purpose∞

Use the SET statement to change session level properties.

Tip

to change properties globally, use the ALTER SERVER statement instead.

Syntax∞

SET <property> { TO | = } { <value> | '<value>' | DEFAULT }

SET CONSTRAINTS { ALL | <constraint> } { IMMEDIATE | DEFERRED }

SET [ SESSION CHARACTERISTICS AS ] TRANSACTION
  {
      ISOLATION LEVEL { READ UNCOMMITTED | READ COMMITTED NO SNAPSHOT | READ COMMITTED | REPEATABLE READ | SNAPSHOT | SERIALIZABLE }
    | { READ ONLY | READ WRITE }
  }

Parameters∞

<property>
The name of the property being changed. See Server Properties for properties that can be changed with SET.
<value>
The new value of the property.
DEFAULT
Restore the property to its default setting.
CONSTRAINTS { ALL | <constraint> } { IMMEDIATE | DEFERRED }
Change when a DEFERRBLE constraint, either all or the one named constraint, are checked. If IMMEDIATE, the constraint(s) are check at the end of the statement. If DEFERRED, they are checked at COMMIT.
SESSION CHARACTERISTICS AS
If supplied when changing a TRANSACTION setting, it stays in effect for the lifetime of the session.
TRANSACTION { READ ONLY | READ WRITE }
If READ ONLY, only statements that read values (e.g. SELECT) may be performed. If READ WRITE, any statement may be performed.
TRANSACTION ISOLATION LEVEL

Change how other, concurrent, transactions affect what data is seen and can be changed by this transaction. The default is SERIALIZABLE, which is the strongest and appropriate for most cases. If transactions concurrently modify data that another transaction reads, only one of the transactions will complete successfully.

READ COMMITTED NO SNAPSHOT allows a read-only transaction to see rolling snapshots of data that has been successfully committed, without encountering errors due to the time limit on how long a transaction can be open. READ COMMITTED NO SNAPSHOT implies READ ONLY.

SNAPSHOT does not generate any read conflicts. Transactions will conflict only because they both try to write the same row, not because one updates what the other reads.

Requesting any other isolation level will actually give the next stronger level that is actually supported. So, REPEATABLE READ will actually give SNAPSHOT.

Examples∞

Setting the encoding the client will send data in:

SET client_encoding TO latin1;

Disallow writes from this transaction:

SET TRANSACTION READ ONLY;

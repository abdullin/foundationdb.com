FoundationDB 3.0 » Operations »

Known Limitations∞

FoundationDB has limitations, some of which may be addressed in future versions. If any of these limitations are particularly painful to you, please write us at support@foundationdb.com so that we can target them in the future.

For related information, also see:
  • Platform Issues that affect the operation of FoundationDB.
  • System requirements for OS/hardware requirements.
  • Core Anti-features for limitations of the scope of the FoundationDB core.
  • Performance considerations for how different workloads can limit performance.

Design limitations∞

These limitations come from fundamental design decisions and are unlikely to change in the short term. Applications using FoundationDB should plan to work around these limitations. See Core Anti-features for related discussion of our design approach to the FoundationDB core.

Long transactions∞

FoundationDB currently does not support transactions running for over five seconds. In particular, after 5 seconds from the first read in a transaction:

  • subsequent reads that go to the database will usually raise a past_version error (although reads cached by the client will not);
  • a commit with any write will raise a past_version or not_committed error.

Clients need to avoid these cases. For the design reasons behind this limitation, see the discussion in Core Anti-features.

Workarounds

The effect of long and large transactions can be achieved using short and small transactions with a variety of techniques, depending on the desired behavior:

  • If an application wants long transactions because of an external process in the loop, it can perform optimistic validation itself at a higher layer.
  • If it needs long-running read snapshots, it can perform versioning in a layer.
  • If it needs large bulk inserts, it can use a level of indirection to swap in the inserted data quickly.

As with all data modeling problems, please ask for help on the community site (or via e-mail) with your specific needs.

Large transactions∞

Transaction size cannot exceed 10,000,000 bytes of affected data. Keys, values, and ranges that you read or write are all included as affected data. Likewise, conflict ranges that you add or remove (using a snapshot read or a transaction option) are also added or removed from the scope of affected data.

If any single transaction exceeds one megabyte of affected data, you should modify your design. In the current version, these large transactions can cause performance issues and database availability can (briefly) be impacted.

Workarounds

See the discussion in long transactions for applicable workarounds.

Large keys and values∞

Keys cannot exceed 10,000 bytes in size. Values cannot exceed 100,000 bytes in size. Errors will be raised by the client if these limits are exceeded.

Workarounds

FoundationDB provides efficient ways to design keys and values to work around this limitation.

Spinning HDDs∞

FoundationDB is only designed for good performance with rotational disk drives when using the durable memory storage engine. It is not recommended that you run FoundationDB on rotational HDDs when using the ssd storage engine. Many algorithms and optimizations have been made to specifically target good performance on solid-state storage that do not translate well to HDDs. Reduced performance and/or database availability issues can be expected.

Recommendation

Large disk arrays and abstracted storage subsystems with sufficient I/O performance may be able to overcome this limitation, but testing specific use cases will be required.

Key selectors with large offsets are slow∞

The current version of FoundationDB resolves key selectors with large offsets in O(offset) time. A common misusage of key selectors is using offsets to page through a large range of data (i.e. reading ‘a’+0 to ‘a’+100, then ‘a’+100 to ‘a’+200, etc.)

Workarounds

An efficient alternative is to use the limit parameter with range reads, starting subsequent reads at the key after the last one returned. You can also use the iterator functionality available in most of the language bindings, which uses this technique internally.

The RankedSet layer provides a data structure in which large offsets and counting operations require only O(log N) time. It is a good choice for applications such as large leaderboards that require such functionality.

Current limitations∞

These limitations do not reflect fundamental aspects of our design and are likely be resolved or mitigated in future versions. Administrators should be aware of these issues, but longer-term application development should be less driven by them.

Cluster size∞

FoundationDB has undergone performance testing and tuning with clusters of up to 500 cores/processes. Significantly larger clusters may experience performance bottlenecks leading to sub-linear scaling or related issues.

Recommendation

If you are thinking of testing on a cluster larger than 500 cores/processes, we encourage you to contact us so we can help you configure machine classes for best results.

Database size∞

FoundationDB has been tested with databases up to 100 TB (total size of key-value pairs – required disk space will be significantly higher after replication and overhead). If you are thinking of using a database larger than 10 TB, please contact us for configuration recommendations.

Recommendation

The main limitation here is our testing environment, so if you are interested in and have the budget for a larger SSD-backed system, please contact us so we can work together on the necessary testing.

Slow machines∞

In some cases, one or more machines that are significantly slower than others in a cluster may reduce the throughput of the entire cluster.

Workarounds

This limitation applies when a hardware component on a server, e.g., a CPU or SSD, has a very low throughput. Administrators should avoid including machines in a cluster that have much slower hardware than others and should watch for machines that report much higher utilization than others. Due to FoundationDB’s fault tolerance, this condition can always be resolved simply by shutting down the offending machine(s) via the administration console.

Multi-datacenter capacity∞

When operating in a multi-datacenter redundancy mode, a datacenter failure is treated in the same manner as the failure of an individual machine, triggering a full replication of data marked as lost. As a result, the storage needs in the remaining datacenter can be instantly doubled. Running out of storage space in this situation can lead to database unavailability.

Workarounds

  • In the current version of FoundationDB, it is best practice to keep three_datacenter systems below 60% disk-space utilization at all times.
  • Multiple-datacenter support in this release is immature. If you are considering using multiple datacenters in production, we recommend that you contact us so we can work together for best results.

Write-scalability with multiple datacenters∞

When operating in a multi-datacenter redundancy mode, you cannot have more than two log servers per datacenter. This restriction limits the system’s ability to scale write throughput for large clusters in multiple datacenters.

No Zero-Downtime Upgrades∞

Upgrades require installing the new version of FoundationDB on each machine in the cluster. As machines are updated, the cluster will become unavailable until enough machines have been upgraded.

Workarounds

If you are using the ssd storage engine (which starts up very quickly) and orchestrate the upgrade process carefully (please contact us for details), the downtime will be only a few seconds, similar to that of certain failure recovery scenarios. A more casually performed upgrade may cause minutes of downtime.

Limited read load balancing∞

FoundationDB load balances reads across the servers with replicas of the data being read. However, it does not currently increase the replication factor of keys that are frequently read. As a result, the aggregate read performance of any given key or small contiguous set of keys in a triple-replicated system is limited to the total performance of three server processes (typically on the order of 100,000 reads per second).

Workarounds

If data is accessed exceptionally frequently, an application could avoid this limitation by storing such data in multiple subspaces, effectively increasing its replication factor. If you are considering this workaround, please contact us for guidance.

Running out of disk space∞

Common low-disk conditions in FoundationDB are handled gracefully via transaction throttling. However, if disk space is completely exhausted (e.g. by another application), database availability can be compromised.

Workarounds

In such extreme cases, deleting data from the database or adding new machines may not suffice to solve the problem because these operations require transaction logging that may itself consume disk space. Administrative action to free space will be required.

Slow return of space to filesystem∞

The ssd storage engine can be slow to return space from deleted data to the filesystem when the number of free pages is very large. This condition can arise when you have deleted most of the data in your database or greatly increased the number of processes in your cluster. In this condition, the database is still able to reuse the space, but the disk files may remain large for an extended period.

Workarounds

Because the database is able to reuse the space, action is required only if your application requires the space for some non-database purpose. In this case, you can reclaim space by excluding affected server processes (either singly or a few at a time) and then including them again. Excluding removes the storage server and its data files completely, and when the server is included again, it uses a fresh database file.

CPU starvation∞

FoundationDB relies on continuous low-latency communication with clients and other processes. If a CPU is highly utilized by other applications, the FoundationDB process can be starved. In this case, the failure detection algorithms can misinterpret non-responsiveness as a machine failure, causing extra data movement and work.

Workarounds

If FoundationDB server processes coexist on a machine with multithreaded or multiprocess workloads that may suddenly have very high load, consider using operating system affinity features to reserve cores for FoundationDB.

Very high-latency networks∞

FoundationDB includes several internal timeouts which can interfere with cluster availability and performance when operating on very high latency networks.

Recommendation

If packet delivery times between nodes in the cluster are approaching one second, consider a different network design. (Sorry, spaceships.)

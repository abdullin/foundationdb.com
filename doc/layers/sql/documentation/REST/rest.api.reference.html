REST API Reference∞

The REST API provides a document style interface to SQL Layer data. It conforms to the same fully-modeled schema and is particularly adapted to access Table Groups data as JSON objects.

This document describes version 1 (/v1) of the REST API. New releases may increase the version number.

In this guide we use the word entity to describe a Table-Group instance.

Connecting to the REST API with curl∞

The cURL utility is a command line tool for transferring data with URL syntax. It can be used for command-line interaction with the SQL Layer’s REST Object API on port 8091. As an example, we will issue a request with curl against the REST API version end point to retrieve the version of SQL Layer running:

$ curl http://localhost:8091/v1/version

If everything is working correctly, you should see output similar to:

[
{"server_name":"FoundationDB SQL Layer","server_version":"2.0.0-SNAPSHOT.4489-1179823"}
]

Note

For all state-changing requests (i.e. not GET requests), the referer header must be set in order to prevent cross-site request forgery attacks. By default the referer must be http://localhost, but this can be changed in the configuration settings. Setting this with curl just requires the addition of the -e option, for example:

$ curl -e http://localhost -X DELETE http://localhost:8091/v1/entity/getting_started_rest.orders/101

API Reference∞

Entity Resources∞

Names and Identifiers∞

A SQL Table-Group is referred to as an entity in REST, where the name of the entity is the fully-qualified (i.e. schema and table) table name and the identifier is the value(s) of the PRIMARY KEY. If basic authorization is enabled for HTTP connections, the schema name defaults to the user name and may be omitted from REST requests.

The representation in an entity identifier in a URL is the same as in SQL for most data types. For example, SELECT * FROM schemaA.example WHERE id=1 is equivalent to GET /v1/entity/schemaA.example/1. Data types that contain non-URL safe bytes, such as generic binary data, may be specified in encoded forms.

For example, assume the given table and row containing the binary values of ASCII ‘ABC’ (decimal values 65, 66 and 67):

> CREATE TABLE schemaA.example(id CHAR(12) FOR BIT DATA NOT NULL PRIMARY KEY, num INT);
> INSERT INTO schemaA.example VALUES (x'414243', 100);

The identifier can be specified as a string (UTF-8 byte values restricted to URL safe range):

> GET /v1/entity/schemaA.example/ABC
[
{"id":"QUJD","num":100}
]

Or as URL-encoded values (again, restricted to URL safe range):

> GET /v1/entity/schemaA.example/%41%42%43
[
{"id":"QUJD","num":100}
]

Or as a hex-encoded string (prefixed with hex:):

> GET /v1/entity/schemaA.example/hex:414243
[
{"id":"QUJD","num":100}
]

Or as a base64-encoded string (prefixed with base64:):

> GET /v1/entity/schemaA.example/base64:QUJD
[
{"id":"QUJD","num":100}
]

PATCH∞

PATCH /v1/entity/{name}∞

Update individual portions of an entity or, if the entity does not exist, create a new one. Only the fields specified in the incoming document will be modified. This provides upsert or merge style behavior.

Input may be a JSON object or array. Each object must contain the full ID of the entity to modify (or create). The given objects may not contain changes to collections. Successful response is a JSON object or array, if one was given, containing the identity for the affected entity or entities.

Parameters:
  • name – name of the entity.

Request

PATCH /v1/entity/employees HTTP/1.1
Content-Type: application/json

{
  "emp_no": 1,
  "first_name": "Clark"
}

Response

HTTP/1.1 200 OK
Content-Type: application/json

{
  "emp_no": 1
}

DELETE∞

DELETE /v1/entity/{name}/{id}∞

Destroy an instance of an entity. Successful response has an empty body.

Parameters:
  • name – name of the entity.
  • id – identifier of the entity.

Request

DELETE /v1/entity/employees/3 HTTP/1.1

Response

HTTP/1.1 204 No Content

Procedure Call Resources∞

For example responses, assume the procedure below has been created on the FoundationDB SQL Layer.

CREATE PROCEDURE js_concat(IN x CHAR(32), IN y CHAR(32), OUT ret CHAR(64))
LANGUAGE javascript PARAMETER STYLE variables AS $$
    ret = x + ' ' + y;
$$;
POST /v1/call/{procedure}∞

Retrieve the results from running the given procedure. Successful result is a JSON array.

Request JSON Object:
 
  • (varies) – Passed directly to the procedure. For the example, x and y are required query parameters.

Request

POST /v1/call/js_concat HTTP/1.1

{
    "x" : "hello",
    "y" : "world"
}

Response

HTTP/1.1 200 OK
Content-Type: application/json

{
  "ret": "hello world"
}

SQL Resources∞

POST /v1/sql/query∞

Run a single SQL statement. Successful response is a JSON array results.

Request JSON Object:
 
  • q (string) – SQL query to run. required

Request

POST /v1/sql/explain HTTP/1.1
Content-Type: application/json

{
    "q" : "SELECT COUNT(*) emp_cnt FROM emp.employees"
}

Response

HTTP/1.1 200 OK
Content-Type: application/json

[
  {
    "emp_cnt": 3
  }
]
POST /v1/sql/explain∞

View the explain plan for running the given query. Successful response is a JSON object containing elements for all pieces of the plan.

Request JSON Object:
 
  • q (string) – SQL query to explain. required

Request

POST /v1/sql/explain HTTP/1.1
Content-Type: application/json

{
    "q" : "SELECT COUNT(*) emp_cnt FROM emp.employees"
}

Response

HTTP/1.1 200 OK
Content-Type: application/json

{
  "type" : "project",
  "input_operator" : [ {
    "type" : "count_operator",
    "name" : [ "Count_TableStatus" ],
    "input_type" : [ {
      "type" : "rowtype",
      "name" : [ "emp.employees" ],
      "table_schema" : [ "emp" ],
      "table_name" : [ "employees" ]
    } ],
    "output_type" : [ {
      "type" : "rowtype",
      "name" : [ "values([MCOMPAT_ BIGINT(21) NOT NULL])" ]
    } ]
  } ],
  "projection" : [ {
    "type" : "field",
    "position" : [ 0 ],
    "name" : [ "Field" ]
  } ],
  "cost" : [ "rows = 3, cost = 7.18640" ],
  "name" : [ "Project_Default" ]
}
POST /v1/sql/execute∞

Execute multiple SQL statements within a single transaction. Individual statements can be free-form and must be separated by semicolons (;). Successful response is a JSON object with one field per statement executed.

Request

POST /v1/sql/execute HTTP/1.1
Content-Type: text/plain

SELECT SUM(salary) total_sal FROM emp.salaries;
SELECT first_name,dept_name
  FROM emp.employees
  INNER JOIN emp.dept_emp USING(emp_no)
  INNER JOIN emp.departments USING(dept_no);
SELECT MIN(from_date) min_title_date FROM emp.titles;

Response

HTTP/1.1 200 OK
Content-Type: application/json

{
  "result_set": [
    {
      "total_sal": 125945
    }
  ],
  "result_set_1": [
    {
      "first_name": "John",
      "dept_name": "Production"
    },
    {
      "first_name": "Sally",
      "dept_name": "Development"
    }
  ],
  "result_set_2": [
    {
      "min_title_date": "1986-06-26"
    }
  ]
}

Version Resource∞

GET /v1/version∞

Request the version of the SQL Layer. This is often used as the “Hello World” of the REST API. Successful response contains a single JSON array with a single object containing the name and version of the FoundationDB SQL Layer.

Request

GET /v1/version HTTP/1.1

Response

HTTP/1.1 200 OK
Content-Type: application/json

[
  {
    "server_name": "FoundationDB SQL Layer",
    "server_version": "1.5.2-SNAPSHOT.2617"
  }
]

View Resource∞

This resource exposes views that have been created through SQL. For demonstration purposes, assume this view has been created on the FoundationDB SQL Layer.

CREATE VIEW name_dept(emp_name,dept_name) AS
    SELECT first_name || ' ' || last_name,dept_name FROM emp.employees
    INNER JOIN emp.dept_emp USING(emp_no) INNER JOIN emp.departments USING(dept_no);
GET /v1/view/{name}∞

Retrieve the results from running the given view. Successful result is a JSON array.

Query Parameters:
 
  • (varies) – Passed directly to the view. For the example, emp_name and dept_name are valid query parameters.

Request

GET /v1/view/name_dept HTTP/1.1

Response

HTTP/1.1 200 OK
Content-Type: application/json

[
  {
    "emp_name": "John Butler",
    "dept_name": "Production"
  },
  {
    "emp_name": "Sally Green",
    "dept_name":"Development"
  }
]

Configuration∞

This document contains reference information for configuring an SQL Layer instance. We recommend that you read this document before setting up a instances for performance testing or production use.

Note

In FoundationDB, a “SQL Layer instance” refers to one SQL Layer process installed on one physical or virtual machine. It is possible to deploy multiple SQL Layer instances against a single FoundationDB Key-Value cluster, and each SQL Layer instance can have it’s own configuration.

To configure a SQL Layer instance, you need to understand some basic aspects of the system. You can start by reviewing the system requirements. Next, review the directory structure and the main configuration files. Finally, you should look at the available configuration parameters, including the logging options, JVM options, and most importantly, the server options, which control the most important aspects of the SQL Layer behavior.

You should also look at the recommended physical architecture, and the installation guides for setting up a SQL Layer cluster.

System requirements∞

Operating systems∞

The SQL layer requires one of the following 64-bit operating systems:

  • Ubuntu 10.04 or later

    • Note: FoundationDB requires Ubuntu 11.10 or later

  • RHEL/CentOS 6

    • Note: FoundationDB requires RHEL/CentOS 6.x

  • Windows XP or later

  • Mac OS X 10.7 or later

Warning

The Mac OS X version of the FoundationDB server and SQL Layer are intended for use on locally accessible development machines only. Other uses are not recommended.

Kernel Requirements∞

  • For RHEL/CentOS

    • 2.6.18 or later

  • For Ubuntu

    • 2.6.32 or later

To determine whether the required kernel is installed, enter the following command:

$ uname -r

The following is a sample output displayed by running the command on a Ubuntu 14.04 system:

3.31.0-24-generic

In this example, the output shows the kernel version (3.13.0) and errata level (24-generic) on the system.

Package Requirements∞

The following are the packages required for the SQL Layer:

  • Java
  • FoundationDB client

Full installation instructions can be found in the Getting Started guides.

Physical Requirements∞

The following are the CPU and memory requirements for installing the SQL Layer:

  • 64 bit processor
  • Default of 512 MB of physical RAM is allocated per SQL Layer instance. We recommend that you increase that to at least 1GB for production deployments.
  • Between 50 MB and 100 MB of disk space for the SQL Layer software depending on what components are installed

To determine the physical RAM available on a Linux system, enter the following command:

$ grep MemTotal /proc/meminfo

To determine the amount of free disk space on a Linux system, enter the following command:

$ df -h

The FoundationDB SQL Layer has no space requirements for data storage because it is stateless and relegates all data storage to the Key-Value cluster. It may require temporary local space for sorting; depending on your data size and query complexity, this may be significant.

Directory Structure∞

Linux Directory Structure∞

The default directory structure created by the package after installation is:

Directory Name Purpose
/etc/foundationdb/sql/* Configuration file storage (see Configuration Files)
/usr/sbin/fdbsqllayer SQL Layer start script
/usr/bin/fdbsql* CLI tool scripts
/usr/share/foundationdb/sql/* Server, client and dependency jar files
/etc/init.d/fdbsqllayer System start script
/var/log/foundationdb/sql/* Log files
/var/run/fdb-sql-layer.pid Process ID file

Note

  • As part of the package install, a foundationdb user is also created. The SQL Layer instance will always run as the foundationdb user when installed from packages.
  • The FoundationDB SQL Layer Java process entry point is com.foundationdb.sql.Main

Max OS X Directory Structure∞

The default directory structure created by the package after installation is:

Directory Name Purpose
/Library/LaunchDaemons/com.foundationdb.layer.sql.plist System start script
/usr/local/etc/foundationdb/sql/* Configuration files (see Configuration Files)
/usr/local/libexec/fdbsqllayer SQL Layer start script
/usr/local/bin/fdbsql* CLI tool scripts
/usr/local/foundationdb/sql/* Server, client and dependency jar files
/usr/local/foundationdb/logs/sql/* Log files
/var/run/com.foundationdb.layer.sql.pid Process ID (pid) file

Windows configuration and data paths∞

The SQL Layer Windows installer creates the CSIDL_COMMON_APPDATA\foundationdb\sql directory for configuration files. The location of this directory can vary depending on the version and language of Windows:

  • On English Windows Vista/7 and newer: C:\ProgramData\foundationdb
  • On English Windows XP/Server 2003: C:\Documents and Settings\All Users\Application Data\foundationdb
  • On non-English Windows varieties, these paths may be localized.

The log files are in CSIDL_COMMON_APPDATA\foundationdb\logs\sql by default, so in Windows 7 they will be in C:\ProgramData\foundationdb\logs\sql\

Configuration Files∞

Several configuration files are installed as part of the FoundationDB SQL Layer package in the configuration directory (See above for the configuration directory path in different OS.) You will see at least these four files:

File Description
log4j.properties Logging configuration
jvm.options JVM configuration
server.properties SQL Layer server configuration
sqllayer.policy SQL Layer policy file

The different options for each of the three configuration files are summarized below. The most important one to review is server.properties.

Logging Options∞

The FoundationDB SQL Layer uses log4j, a common Java logging library. There are multiple log4j properties that determine what gets logged and how. The important logging configuration properties are:

Property Description Default
log4j.appender.rollingFile.File Name of file to rollover to when logging layer.log (under Directory Structure logging)
log4j.appender.rollingfile.MaxFileSize maximum log file 10MB
log4j.appender.rollingfile.MaxBackupIndex number of log files 10

Note

log4j system logging is different from query logging. The log4j logs hold the SQL instance general log that contains a trace of all activity in the local SQL Layer, including startup, shutdown, session info, errors, etc. When enabled, Query logging captures only queries that exceed the exec_threshold_ms threshold.

JVM Options∞

The SQL Layer runs on the Java Virtual Machine (JVM), which has quite a large number of options. Below are the important options we include in the jvm.options file distributed with SQL Layer along with their default values. Other JVM options are available in Oracle’s Java website.

JVM Option/Property Description Default
MAX_HEAP_SIZE Maximum heap size 512M

Server Properties∞

Server properties control the behavior of the SQL Layer instance. They are set in the server.properties configuration file on startup.

A few of these properties can be set at runtime. These variables are identified in the tables below where Dynamic property is YES. They can be updated using the SET command for setting Session scope or the ALTER SERVER statement for setting Global scope. To change variables that can not be set at runtime, proceed to change the configuration file and restart the SQL Layer.

The name of the SET property is the configuration property without its dotted prefix. So, for fdbsql.sql.binary_output='hex' in the file, the corresponding statement is SET binary_output TO 'hex'.

Cluster Options∞

Cluster options control the connection parameters the SQL Layer uses to connect to the underlying FoundationDB Cluster. See more information regarding setting up a cluster in the installation guides.

  • fdbsql.fdb.cluster_file
fdbsql.fdb.cluster_file∞

Path to the FoundationDB Key-Value cluster file. This file must exist and be readable by the user starting the SQL Layer. See Cluster files form more details.

Property type String
Default value See Default cluster file
Dynamic property No
Range of values Any path writable by SQL Layer

fdbsql.default_charset∞

Specifies the default charset used. It is possible to override by setting a different charset in the CREATE TABLE statement. Supported charsets are listed in the charset section of the Localization page.

Property type String
Default value utf8
Dynamic property No
Range of values Any supported charset
fdbsql.default_collation∞

Specifies the default collation used. It is possible to override by setting a different collation in the CREATE TABLE statement. Supported collations are listed in the collation section of the Localization page.

Property type String
Default value ucs_binary
Dynamic property No
Range of values Any supported collation
fdbsql.postgres.host∞

Host on which to start the Postgres Server.

Property type String
Default value localhost
Dynamic property No
Range of values IP | hostname
fdbsql.postgres.port∞

Port on which to start the Postgres Server.

Property type Integer
Default value 15432
Dynamic property No
Range of values 1 .. 65535
fdbsql.postgres.OutputFormat∞

Controls whether Nested SQL query format is enabled, and therefore wether results are returned in table format or JSON format. Nested SQL Queries with nested select clauses are supported only when the output format is set to JSON.

Property type String
Default value table
Dynamic property Yes
Range of values table | json
fdbsql.postgres.statementCacheCapacity∞

Controls the FoundationDB SQL Layer statement cache. If a query uses bind variables or a query with the same literals is sent to the FoundationDB SQL Layer, the statement cache can be used. Any statement retrieved from the cache avoids the need to perform query optimization since an execution plan for the query is kept in the statement cache.

Property type Integer
Default value 500
Dynamic property Yes
Range of values 0 .. 4096
fdbsql.routines.script_class_path∞

Specifies a path for additional script engines. The SQL Layer will check these paths and register script engines. Routines are then able to use the registerd names as a script language.

Property type String
Default value “”
Dynamic property No
Range of values Any path readable by SQL Layer
fdbsql.sort.memory∞

Controls the size of the memory buffer allocated for sorting operations. If the data exceeds this size, the buffer will be written to disk, in the fdbsql.tmp_dir directory, with as many files as needed to sort the data. Defaults to 64M.

Property type Integer
Default value 64MB
Dynamic property No
Range of values 0 .. OS Limit
fdbsql.sql.binary_output∞

Specifies the format to be used when printing binary data as SQL columns.

Property type String
Default value octal
Dynamic property Yes
Range of values octal | hex | base64
fdbsql.sql.constraintCheckTime∞

Controls when uniqueness constraints are checked when inserting or updating rows. An important special case of a uniqueness constraint is the primary key. Every table has a primary key, even if none is declared. STATEMENT performs all checks for each statement in parallel, which can improve performance when multiple rows are being inserted / updated. DELAYED does not perform checks until COMMIT time, which gives maximum performance, but at the cost of potentially inconsistent SELECT behavior in a transaction that will later report an error. The _WITH_RANGE_CACHE variants also do a snapshot range scan, which can prefill the transaction cache with knowledge that there are no keys in a sequential range. See the Performance Guide for more general information on using this setting to improve bulk insert performance.

Property type String
Default value IMMEDIATE
Dynamic property Yes
Range of values IMMEDIATE | STATEMENT | DELAYED | STATEMENT_WITH_RANGE_CACHE | DELAYED_WITH_RANGE_CACHE
fdbsql.sql.jsonbinary_output∞

Specifies the format to be used when printing binary data as JSON values.

Property type String
Default value base64
Dynamic property Yes
Range of values octal | hex | base64
fdbsql.sql.queryTimeoutSec∞

The threshold in seconds for the time to allow a query to run. If a query is executing for longer than this threshold, it will be aborted. Set this to 0 to disable the threshold.

Property type Double
Default value 0
Dynamic property Yes
Range of values Any positive double
fdbsql.sql.statement_timeout∞

The threshold in milliseconds for the time to allow a query to run. If a query is executing for longer than this threshold, it will be aborted. Set this to 0 to disable the threshold.

Property type Integer
Default value 0
Dynamic property Yes
Range of values Any positive integer
fdbsql.tmp_dir∞

Path to location where temporary files are created during some sort operations and other local processing. default /tmp (really?) Temp file are created during some sort operations and other local processing. This directory need not exist before starting the server however the user starting the SQL Layer instance will need correct permissions in order to create the directory.

Property type String
Default value System default (e.g. /tmp, %TEMP%)
Dynamic property No
Range of values Any path writable by the SQL Layer process
optimizer.implicitAggregate∞

Controls how non-aggregate fields in an aggregate query are handled. The default value ‘error’ is compatible with the SQL standard. ‘first’ returns the first value always, and ‘firstIfUnique’ returns the first value if that is still deterministic.

Property type String
Default value error
Dynamic property Yes
Range of values error | first | firstIfUnique

The following options control the handling of requests from clients operating in a different domain than the SQL Layer is listening on. This is commonly referred to as Cross-Site Request Access Control:

  • fdbsql.http.cross_origin.allow_credentials
  • fdbsql.http.cross_origin.allowed_headers
  • fdbsql.http.cross_origin.allowed_methods
  • fdbsql.http.cross_origin.allowed_origins

fdbsql.http.cross_origin.allow_credentials∞

A boolean indicating if the resource allows requests with credentials.

Property type boolean
Default value true
Dynamic property No
Range of values true | false
fdbsql.http.cross_origin.allowed_headers∞

Comma separated list of HTTP headers that are allowed to be specified when accessing resources.

Property type String
Default value X-Requested-With,Content-Type,Accept,Origin
Dynamic property No
Range of values Comma separated list of HTTP headers
fdbsql.http.cross_origin.allowed_methods∞

Comma separated list of HTTP verbs that are allowed to be used when accessing resources. For more information on end points see the REST API reference.

Property type String
Default value HEAD,GET,PUT,POST,DELETE,PATCH
Dynamic property No
Range of values Comma separated list of HTTP methods
fdbsql.http.cross_origin.allowed_origins∞

Comma separated list of origins that are allowed to access the resources.

Property type String
Default value http://localhost:8091
Dynamic property No
Range of values Comma separated list of origins
fdbsql.http.cross_origin.enabled∞

Determines whether Cross-Site requests are allowed. If set to false, the rest of the cross_origin configuration options are ignored, and no Access-Control-Allow-* headers are sent. If set to true, Access-Control-Allow-* headers are sent as specified by the other cross_origin options, allowing for cross-origin requests.

Property type boolean
Default value false
Dynamic property No
Range of values true | false
fdbsql.http.cross_origin.preflight_max_age∞

The number of seconds that preflight requests can be cached by the client.

Property type integer
Default value 1800
Dynamic property No
Range of values Any integer

The following options handle how CSRF protection is applied.

  • fdbsql.http.csrf_protection.allowed_referers
  • fdbsql.http.csrf_protection.type
fdbsql.http.csrf_protection.allowed_referers∞

Comma separated list of referers that are allowed to access the resources. Only scheme, domain and port should be provided. Username, password, path and anchor are prohibited.

Property type String
Default value http://localhost
Dynamic property No
Range of values Comma separated list of URIs
fdbsql.http.csrf_protection.type∞

How the CSRF protection will be applied.

none
Disables all csrf protection.
referer
Verifies that the referer header is from one of the origins in allowed_referers. GET requests without a referer header or with a blank referer header are also allowed. All other requests prohibit blank or missing referer headers.
Property type String
Default value referer
Dynamic property No
Range of values none | referer

Query Logging Options∞

The SQL Layer provides an option to log queries issued to the server into a local file. The query log can be configured to log all queries or only queries taking longer than a configured time threshold to execute.

Note

Query logging is different than the general logging of the SQL Layer.

The query log file is a plain text, structured file that is readable by humans and can easily be parsed by tools. Here’s an example of some output in the query log file:

#
# 2011-09-08 22:39:40.727
# session_id=6
# execution_time=53391
# rows_processed=2139
SELECT name, oid FROM customers INNER JOIN orders ON customers.cid=orders.cid WHERE order_date < '2014-02-02'
#
# 2011-09-08 22:40:06.63
# session_id=6
# execution_time=17782
# rows_processed=5000
SELECT * FROM orders
#
# 2011-09-08 22:40:06.63
# session_id=6
# execution_time=12
# error_msg=com.foundationdb.server.error.NoSuchColumnException: NO_SUCH_COLUMN: Unknown column: `no_such_table`
SELECT no_such_table

Events are separated by a single ‘#’ character. This record separator can be used by scripts to easily separate events in a query log file. The separator marks the end of the event.

Possible lines for an event are:

  • Timestamp at which the query was logged (roughly query start time + execution_time) in the timezone of the logging SQL Layer instance.
  • session_id - Identifier of the executing session.
  • execution_time - Query duration in milliseconds.
  • rows_processed - (Optional) Number of rows processed by the query.
  • error_msg - (Optional) Error class, code, and message for failing statements.

Note

In a distributed configuration, each SQL Layer instance will have a local log file containing queries that ran on that instance only.

Several properties in the server.properties file control query logging:

  • fdbsql.querylog.enabled
  • fdbsql.querylog.exec_threshold_ms
  • fdbsql.querylog.filename
fdbsql.querylog.enabled∞

Whether the FoundationDB SQL Layer query log is enabled or not.

Property type boolean
Default value false
Dynamic property Yes
Range of values true | false
fdbsql.querylog.exec_threshold_ms∞

The threshold in milliseconds for the time to log a query in the query log. If a query is executing for longer than this threshold, it will be logged in the query log file. Set this to 0 to log all queries issued against the SQL Layer.

Property type Integer
Default value 0
Dynamic property Yes
Range of values Any positive integer
fdbsql.querylog.filename∞

File in which the FoundationDB SQL Layer query log will places queries it logs.

Property type String
Default value queries.log (under Directory Structure logging)
Dynamic property Yes
Range of values Any file writable by the SQL Layer

Policy File∞

The policy file defines permissions for packages. The java security manager will use this as a source to determine whether a certain action can be performend. It limits the capabilities of functions and procedures to perform malicious actions to the system. The routine firewall is the bottleneck for permissions. Permissions can be granted for specific actions using standard Java syntax for policy files.

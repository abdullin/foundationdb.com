Application Integration∞

This section will guide your use of SQL Layer through programming languages, ORMs and other tools.

ORM Support∞

The preferred way to use SQL Layer is through Object-Relational Mapping layers (ORMs).

Several ORMs have been tested with SQL Layer and have specific adapters for it. Some of the implementations simply ensure that the SQL syntax and metadata queries are compatible. Others, like the SQLAlchemy dialect, take advantage of the unique hierarchical modeling and nested SQL querying capabilities of SQL Layer to improve their performance by accessing objects in a single query.

The following ORMs are supported:

  • Java Hibernate

  • Java MyBatis

    • Note: Requires the driverClassName to be set to com.foundationdb.sql.jdbc.Driver, and the url should be of the form: jdbc:fdbsql://host:15432/mybatis-example. See the dataSource section in MyBatis Configuration for more details.

  • node.js Waterline

  • PHP Doctrine

  • Python SQLAlchemy

  • Python Django

  • Ruby ActiveRecord

  • Ruby Sequel

We are always looking to expand the supported list. Please let us know on the community site.

Client Drivers∞

SQL Layer Native Drivers∞

The official JDBC Driver is available from Maven Central.

The official ODBC Driver is available from Get FoundationDB.

PostgreSQL Compatible Drivers∞

SQL Layer “speaks” the PostgreSQL protocol which allows most existing drivers to be used as-is. Almost every language has such a driver and a complete list is maintained at Software Catalogue - Drivers and interfaces. Be sure to read the notes below about untested drivers and ORMs.

Following is a list of drivers that have been tested to work with SQL Layer:

  • C

    • libpq, used by psql client and many of the drivers below

  • CommonLisp

    • CommonSQL (usage example)
    • Postmodern (usage example)

  • Lua

    • pgmoon

  • node.js

    • pg (usage example)

  • PHP

    • PGSQL
    • PDO_PGSQL (usage example)

  • Python

    • psycopg2
    • pg8000
    • Requests (REST access)

  • Ruby

    • pg (usage example)

  • Scala

    • postgresql-async

      • Note: Requires binary_output to be set to hex

  • .NET

    • Npgsql 2.0.13

      • Tested with VisualStudio 11 and .NET 4.5

Troubleshooting Untested Drivers and ORMs∞

It is usually not hard to get an existing driver or binding to work correctly.

The majority of the problems one might encounter are due to how the driver or ORM query metadata. Many PostgreSQL bindings use the System Catalogs directly, whereas SQL Layer only exposes ANSI SQL information_schema tables. Other less common problems include runtime settings, data types, functions, and syntax.

As you are testing a new integration, keep an eye on the log files, where you’ll see reports of failed queries. For example, failures with the error “Cannot find the table pg_xxx” mean that a metadata query needs to be rewritten to use the corresponding information_schema tables. Replace the problem queries in the driver or ORM code and repeat until the system works correctly.

Tools∞

The following tools have been tested to work with SQL Layer:

  • Squirrel SQL
  • IntelliJ IDEA Database Browser
  • RubyMine Database Browser
  • Tableau (setup instructions)

Tools known not to work correctly:

  • pgAdmin III
  • PG Commander

Aggregation∞

Aggregation applies to rows. The incoming rows to this operator are aggregated until a change in the GROUP BY columns, then the row is output. This operator does no sorting or hashing of values. It is known as a partial aggregation operator. To ensure correct aggregation results, it is almost always preceded by a Sort operator.

Behavior∞

This operator takes as input a row which is interpreted as having two sections: a GROUP BY section and an inputs section. These are delimited by the inputs Index argument, which specifies the index of the first input. The operator also has a list of aggregation functions, one per input.

For each input row of input row type , the Aggregation’s cursor applies each input to its appropriate aggregation. When the cursor notices a change in any of the GROUP BY columns or the input cursor is finished, the aggregation cursor outputs a row with the GROUP BY columns and the result of each aggregation.

Example∞

Let’s say we have a table describing various SKUs in warehouses. Each warehouse has a category for SKUs, but these categories are not unique among all warehouses. We want to get the sum, min, and max price of each category in each warehouse.

SELECT warehouse, product_category, SUM(price), MIN(price), MAX(price)
FROM products GROUP BY warehouse, product_category</i>

The input rows to the Aggregation_Partial cursor would be something like:

warehouse product category price price price notes
0001 AAA 5.00 5.00 5.00 sku 1
0001 AAA 10.00 10.00 10.00 sku 2
0002 AAA 7.00 7.00 7.00 sku 3
0002 AAA 3.00 3.00 3.00 sku 4
0002 BBB 11.00 11.00 11.00 sku 5

The input index here is 2. Note that the “price” column has been repeated three times, once for each aggregate function. In this case, the input rows are already ordered by their GROUP BY columns; if we knew this were the case (due to another operator’s ordering), this partial aggregation would also be a full aggregation.

The output rows would look like:

warehouse product category SUM(price) MIN(price) MAX(price)
0001 AAA 15.00 5.00 10.00
0002 AAA 10.00 3.00 7.00
0002 BBB 11.00 11.00 11.00

Notes∞

If there are no input rows, behavior depends on the input index. If it is 0 (no GROUP BY) columns, this Aggregation will not output any rows. If input index is greater than 0 (there is a GROUP BY), a single row will be outputted with all NULL values. This is due to the SQL spec.

This operator cannot do something like an average directly. Instead, the operator tree would use this operator to get a SUM and COUNT of rows, and another operator would be responsible for dividing them to get the average.

Output∞

All input rows are swallowed. Output rows are as described above. All rows from the incoming operator with a type other than input.rowType() are passed through unchanged.

Assumptions∞

A full aggregation requires ordering the rows by their GROUP BY columns.

If an aggregation function is not amenable to this piecemeal aggregation, you should not use it with a partial aggregation.

Performance∞

Partially dictated by aggregators, though expected to be minimal. Comparison of GROUP BY columns is O(N).

Memory requirements∞

One row and one set of grouping column values.

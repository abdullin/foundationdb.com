FoundationDB SQL Layer 2.1 » SQL Layer Developer Reference » SQL Reference » Query Execution »

Update∞

Each row in the input stream is evaluated according to an update function and written into the storage.

Behavior∞

For each row from the input operator’s cursor, it invokes an evaluation of the update Function to create a new version of the row. As a side effect, this new row is updated into the Storage system. The new version of the row is emitted to the output stream. The input row is discarded.

Output∞

The row as modified by the update Function.

Assumptions∞

The input rows are user table rows, for the storage update to process correctly.

Performance∞

Updating rows may be slow, especially since indexes are also updated. There are several random-access reads and writes involved, which depend on the indexes defined for that row type.

There are potentially ways to optimize this, if we can push WHERE clauses down. This would mean we could update some indexes as a batch operation, rather than one at a time. This would require changes to the API, and is not currently a priority.

Memory Requirements∞

Each update function evaluation call method may generate a new row.

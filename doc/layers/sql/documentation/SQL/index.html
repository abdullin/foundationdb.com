SQL Reference∞

The SQL Layer is an ANSI SQL 92 compliant SQL implementation on top of FoundationDB. This document covers:

  • Data Definition Language Statements (DDL)
  • Data Manipulation Language Statements (DML)
  • Configuration and System Management Statements
  • Queries, Algebra and Execution Plans
  • Indexing
  • Data Types, Functions and Localization
  • Information Schema

Note

We recommend reading the Introduction to Table-Groups, and the Getting Started with SQL guide, before exploring this section.

Tip

To interact with a SQL Layer instance use the fdbsqlcli Command Line Tool.

The SQL Language∞

Data Definition Language Statements (DDL)∞

The data definition statements are used to mange SQL objects such as schemas, tables and indexes. The statements below are ANSI standard compliant with several implementing Table-Group specific extensions.

  • ALTER GROUP
  • ALTER TABLE
  • CREATE FUNCTION
  • CREATE INDEX
  • CREATE PROCEDURE
  • CREATE SCHEMA
  • CREATE SEQUENCE
  • CREATE TABLE
  • CREATE VIEW
  • DROP FUNCTION
  • DROP GROUP
  • DROP INDEX
  • DROP PROCEDURE
  • DROP SCHEMA
  • DROP SEQUENCE
  • DROP TABLE
  • DROP VIEW
  • RENAME TABLE
  • TRUNCATE TABLE

Note that there is no CREATE GROUP command because a Table-Groups are implicitly created for any table that does not have a GROUPING FOREIGN KEY. See the Table-Groups guide for more details.

Data Manipulation Language Statements (DML)∞

The data manipulation statements are used to access and mange the contents of SQL objects. The statements below are ANSI standard compliant. Additionally, SELECT is extended with the Nested SQL Query format, which returns hierarchical result-sets by allowing sub-select statements in the SELECT clause to return multiple tuples as a single object.

  • DELETE
  • INSERT
  • SELECT
  • SELECT (Nested)
  • UPDATE

Configuration and System Management Statements∞

The statements below manage the SQL Layer itself along with setting or showing system and session properties.

  • ALTER SERVER
  • SET
  • SHOW

Queries, Algebra and Execution Plans∞

The query execution approach in the SQL Layer is unique in its use of Table-Groups rather than tables as its guiding data structure. The approach and implementation are described in the Query Execution section.

  • Query Execution

Indexing∞

Single and multi-column indexes are supported on most data types. Additionally, Table-Groups support new indexing capabilities in the form of indexes that span multiple tables in a single Table-Group. This section will also cover full-text and geospatial indexes that are integrated into the SQL Layer.

Information Schema∞

An information_schema containing read-only system views is always available to clients and exposes schema and server information. A list and description of all views is below, with full details available by clicking on each view name.

Table Description
ak_casts  
ak_overloads  
columns Columns define the data types of the tuples for each row in a table. Columns have a defined type, size, and ordering within the table. Columns are created when the table is created, and modified by ALTER TABLE.
error_code_classes Lists the varying classes for error codes, along with a brief description. See error code classes for all possible error classes.
error_codes The error messages view shows the defined error code and (locale specific) error messages. Error messages are defined internally and cannot be modified. See error code classes for all possible error codes.
grouping_constraints The grouping_constraints table holds the additional information regarding the Table-Groups GROUPING FOREIGN KEY constraints. There will be one row in this table for each row in the table constraints of ‘GROUPING’.
index_columns Index columns provides the list of column in index.nnGroup indexes have columns from multiple tables. The table_name is the name of the table which is associated with the index, in the INDEXES table. The column_schema_name and column_table_name are for the table in which the column resides. For non-group indexes, these table names will always be the same.
index_statistics Index statistics table contains a histogram of index values for use by the optimizer. It is the first form of statistics gathered by the system. Rows are entered into this table when ALTER TABLE ... UPDATE STATISTICS is executed.
index_statistics_entry Index statistics entries are collected by execution of index analysis. The data is used by the SQL optimizer. Each entry is one histogram bucket.
indexes Indexes are a physical implementation of data storage for improving access. Each of the primary key, unique, and foreign key constraints generate indexes. The grouping foreign key constraint does not. It is possible to create an index that is not related to any constraint and applies to the group (and underlying group table) across the different tables in the group.
jars  
key_column_usage Key column use lists the columns selected for constraints of the type unique, primary key, foreign key or grouping.
parameters  
referential_constraints Referential constraints table hold the additional information required for foreign keys. There should be one row in this table for each row in the table constraints table of type ‘FOREIGN KEY’.
